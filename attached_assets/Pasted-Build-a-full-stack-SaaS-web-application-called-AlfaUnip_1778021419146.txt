Build a full-stack SaaS web application called AlfaUnipac GV — 
Sistema de Gestão Acadêmica for Faculdade AlfaUnipac, campus 
Governador Valadares/MG, Brazil.

---

## IMPORTAÇÃO DE FREQUÊNCIA — EXCEL/SHAREPOINT

The frequency spreadsheet has one sheet per course. Structure:
- Row 1: "Curso: [NOME DO CURSO]"
- Row 2: "Tutor:"
- Row 3: Semester label "2026/1"
- Row 4: Discipline group headers spanning multiple columns 
  (D1, D2, D3, D4...) — each group covers several class dates
- Row 5: Individual date columns under each discipline 
  (format DD/MM) plus "TOTAL DE FALTAS" as last column
- Row 6+: Student data — Nº Matrícula, Nome, 
  then one cell per class date ("f" = absent, "." = present)

Import logic:
1. Read each sheet tab = one course
2. Parse discipline groups from row 4 (D1, D2, D3, D4)
3. Parse dates from row 5 — map each date column to its 
   parent discipline group
4. For each student row: read matricula, name, then for each 
   date column read "f" (absent=true) or "." (absent=false)
5. Upsert into Chamada table: if record exists for 
   aluno+disciplina+data, update; if not, insert
6. After import, recalculate % faltas per student per 
   disciplina and flag anyone above 25% threshold 
   into Retencao table with status "Identificado"

Build TWO import methods:
A) Manual upload button — user uploads .xlsx or .ods file 
   directly in the system
B) Microsoft 365 sync — "Conectar Microsoft 365" button 
   that triggers OAuth2 login with institutional Microsoft 
   account. After login, system stores the refresh token 
   and uses Microsoft Graph API to read the SharePoint file 
   automatically. Add a "Sincronizar agora" button and 
   schedule automatic daily sync at 6am.
   Graph API endpoint to use:
   GET /me/drive/root:/path/to/file:/content
   Accept: application/vnd.openxmlformats-officedocument
   .spreadsheetml.sheet

For Microsoft OAuth2 use these endpoints:
- Auth: https://login.microsoftonline.com/common/oauth2/v2.0/authorize
- Token: https://login.microsoftonline.com/common/oauth2/v2.0/token
- Scopes: Files.Read, offline_access
Store CLIENT_ID and CLIENT_SECRET in environment variables.

## 4. RETENÇÃO

### Roles and permissions
Secretaria: identifies, forwards, fills cancellation 
form, removes from BAP, notifies HBS, reintegrates.
Retencao: contacts student, registers outcome, 
collects signature, returns signed doc.
Coordenacao: signs cancellation form inside system.

### Full status flow (in order)
1. Identificado — auto-flagged by system (>25% absences). 
   Owner: Secretaria.
2. Encaminhado — Secretaria sent to Retencao sector. 
   Owner: Retencao.
3. Em_Contato — Retencao contacted the student. 
   Owner: Retencao.
4. Aguardando_Resposta — waiting for student reply. 
   Owner: Retencao.
5a. Retorno_Confirmado — student decided to continue. 
    Owner: Retencao → returns to Secretaria.
5b. Cancelamento_Solicitado — student decided to leave. 
    Owner: Retencao → returns to Secretaria.
6. Formulario_Preenchido — Secretaria filled cancellation 
   form and sent to Retencao for signature collection. 
   Owner: Retencao.
7. Aguardando_Assinatura — sent to Coordenacao Pedagogica. 
   Owner: Coordenacao.
8. Assinado — Coordenacao signed. Retencao returns signed 
   doc to Secretaria. Owner: Secretaria.
9. Enviado_CRM — Secretaria sent to cancellation sector 
   to remove student from CRM. Owner: Secretaria.
10. Removido_BAP — system automatically removes student 
    from current month BAP list and generates alert: 
    "Aluno [nome] removido da BAP de [mês/ano]. 
    Notificar o Hospital Bom Samaritano."
    Owner: automatic.
11. HBS_Notificado — Secretaria informed Hospital 
    Bom Samaritano. Owner: Secretaria.
12. Encerrado — process complete.

For path 5a (Retorno_Confirmado): 
Secretaria reorients student, status returns to Ativo 
in Aluno table. Retention record status = Reintegrado.

### Main retention screen
Top tabs: Todos / Secretaria / Retenção / 
Coordenação / Encerrados.
Each tab filters by current owner role.
Table columns: Nome, Curso, % Faltas, Etapa atual, 
Responsável, Data última ação, button Ver.
Absence bar: orange 25–35%, red >35%.

### Student detail panel
Horizontal stepper showing all 12 stages with 
current stage highlighted.
Student data: nome, curso, matricula, valor mensalidade.
Disciplines with absences: name, total aulas, 
total faltas, %.
Action timeline: who did what and when.
Action buttons shown based on current status AND 
logged-in role:
- Secretaria sees: Encaminhar para Retenção / 
  Preencher formulário / Retirar do CRM / 
  Remover da BAP / Notificar HBS / Reintegrar aluno
- Retencao sees: Registrar contato / 
  Registrar resposta do aluno / 
  Encaminhar para assinatura / 
  Devolver documento assinado
- Coordenacao sees: Assinar documento (opens modal 
  showing full form for review before signing)

### Cancellation form (auto-generated PDF)
Generated when Secretaria clicks Preencher formulário.
Fields: nome completo, CPF, matrícula, curso, 
data da decisão, motivo declarado, nome da 
coordenadora pedagógica, signature field, 
data assinatura, carimbo institucional.
PDF stored on student record. Status tracked: 
Pendente / Assinado.

### BAP auto-removal
When status reaches Removido_BAP, system automatically 
sets student ativo=false in BAP_Mensal for current month 
and displays alert to Secretaria.

### Summary cards at top
Em acompanhamento (status 2–4), 
Críticos >40% faltas, 
Aguardando assinatura, 
Encerrados no mês.
